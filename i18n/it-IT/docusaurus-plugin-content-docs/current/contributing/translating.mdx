:::info Prerequisites
Before contributing translations, make sure you have:

- Node.js and a package manager installed.
- A local clone of this repository.
- Working knowledge of Git and pull requests.
- Ability to run and verify a local Docusaurus build.
:::

This guide explains how localization is configured in this wiki and how to contribute high-quality translations.

## Localization Model Overview {#localization-model-overview}

This site uses Docusaurus i18n with English as the source locale.

- **Source docs:** `docs/`
- **Localized docs:** `i18n/<locale>/docusaurus-plugin-content-docs/current/`
- **Localized UI strings:** `i18n/<locale>/code.json`, `navbar.json`, `footer.json`, and other generated JSON files.

Configured locales:

- `es-ES`
- `fr-FR`
- `it-IT`
- `de-DE`

## One-Time Locale Setup {#one-time-locale-setup}

1. Add locales in `docusaurus.config.js` under `i18n.locales` and `i18n.localeConfigs`.
2. Generate translation files for each locale:

```bash
npx docusaurus write-translations --locale es-ES
npx docusaurus write-translations --locale fr-FR
npx docusaurus write-translations --locale it-IT
npx docusaurus write-translations --locale de-DE
```

3. Copy English docs into each locale's docs `current/` directory when bootstrapping new docs content.

## Translating Documentation Files {#translating-documentation-files}

When translating `.mdx` files:

- Translate visible text only.
- Keep frontmatter keys, IDs, slugs, links, and code blocks structurally correct.
- Preserve admonitions (`:::note`, `:::tip`, `:::warning`, etc.).
- Keep static heading IDs unchanged (for example, `## Example {#example}`).

:::tip Consistency Rule
Use the same terminology for repeated technical terms across all pages in the same locale.
:::

## Translating UI Strings {#translating-ui-strings}

Translate generated JSON values in locale folders, including:

- `i18n/<locale>/code.json`
- `i18n/<locale>/docusaurus-theme-classic/navbar.json`
- `i18n/<locale>/docusaurus-theme-classic/footer.json`

Do not modify translation keys unless the source key changed.

## Validation Workflow {#validation-workflow}

Run these checks before opening a PR:

```bash
npm run build
npm run start -- --locale <locale>
```

Then verify:

- No broken links.
- No malformed MDX.
- Sidebar categories render correctly.
- Terminology and grammar are consistent.

## Updating Existing Translations {#updating-existing-translations}

When English docs change:

1. Pull latest `main`.
2. Compare source docs against each locale copy.
3. Update only changed sections where possible.
4. Re-run build checks.
5. Open a PR describing which locales and files were updated.

## Pull Request Expectations {#pull-request-expectations}

- Keep PR scope clear by locale and section.
- Include screenshots for visible UI/content changes when practical.
- Note any intentionally untranslated strings and why.
- Ensure translation quality (grammar, spelling, punctuation, and tone) before review.
